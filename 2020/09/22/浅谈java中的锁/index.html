浅谈java中的锁

发表于 2020-09-22 分类于 java 阅读次数：

简介

锁是在多进程并发竞争资源的时候保证数据稳定的手段，宏观的思想是将并行的请求串行化。串行化会引起效率降低，于是便出现了多种多样的锁在应对不同的场景下使得效率尽可能的高。

java中的锁：synchronized，ReentrantLock（线程独占），Semaphore（线程共享），CountDownLatch（线程共享）等等。

synchronized

JDK1.6以前都是重量级锁，锁粒度太大，并发度受到相当大的限制，JDK1.6以后虚拟机中底层优化了锁的机制，对synchronized实现了几种锁，分别是：偏向锁，轻量级锁，重量级锁。一般使用synchronized后，当有新的请求来到，如果有请求正在占用则加入队列排队。锁的升级：偏向锁-》轻量级锁-》重量级锁。java虚拟机默认开启偏向锁和轻量级锁，也可以通过启动参数使用无锁。

重量级锁：重量级锁使用monitor实现，底层使用了操作系统的mutex lock来实现，所以也称互斥锁。操作系统的线程切换需要从用户态转换到内核态，成本很高，在jdk1.6以前只有这一种锁，为了减少获得锁和释放锁带来的性能消耗，引入了轻量级锁和偏向锁。

轻量级锁：适用于没有多线程竞争，线程交替执行同步代码区的情况。一旦同一时间有多个线程同时请求同一把锁的时候就会膨胀为重量级锁。轻量级锁会有多次CAS操作，因为多线程会竞争锁。加锁过程：

（1）判断锁状态是否为无锁，如果是就将当前线程的栈帧建立一个锁记录（Lock Record），存储锁对象（即我们使用synchronized时指定的对象）的mark word，官方称为displaced mark word（因为原来的mark word会被替代）

（2）拷贝对象头的mark word复制到线程的栈帧的锁记录中

（3）使用CAS操作将对象头的mark word更新为指向锁记录的指针（就在此时被替代了，猜测是为了让其他线程判断是否有线程已经获取锁）

（4）如果（3）更新成功，则当前线程就拥有了锁对象，可以进入同步代码区

（5）如果（3）更新失败，判断锁对象头的mark word是否指向自己，如果是则说明拥有锁对象，可以进入同步代码区，如果不是则说明存在多个线程竞争锁，需要升级为重量级锁

偏向锁：适用于单线程的场景，偏向指的是会偏向第一个获得偏向锁的线程，如果再次访问和退出同步代码区的时候不会进行加锁和解锁操作，而是做以下的操作：
（1）load-and-test，判断当前线程id与markword中的线程id是否一致，一致说明成功获得了锁继续执行代码，不一致执行（2）
（2）检查对象是否还可偏向，未偏向则执行CAS操作来竞争锁，也就是第一次获取锁时的操作；如果已经偏向，而且不是偏向自己，则说明存在竞争，升级为轻量级锁

适应性自旋（Adaptive Spinning）：该锁设计的目的是为了不加入队列排队，认为被锁的代码区可能已经被访问完，只是由于线程上下文切换的原因没有及时的让出资源，通过循环占用cpu消耗无意义的时间来等待。出现的时间点：当线程在获取轻量级锁的过程中执行CAS操作失败时，是要通过自旋来获取重量级锁的。JDK中采用与普通自旋不同的操作，如果自旋成功了，下次可自旋次数增加，如果自旋失败了，下次自旋次数减少。自旋是为了让轻量级锁不升级为重量级锁，一旦升级为重量级锁就不再会有自旋操作。JDK1.5之前的版本是固定次数的自旋，JDK1.6之后改为适应性自旋。

（1）如果平均负载小于CPUs则一直自旋
（2）如果有超过(CPUs/2)个线程正在自旋，则后来线程直接阻塞
（3）如果正在自旋的线程发现Owner发生了变化则延迟自旋时间（自旋计数）或进入阻塞
（4）如果CPU处于节电模式则停止自旋
（5）自旋时间的最坏情况是CPU的存储延迟（CPU A存储了一个数据，到CPU B得知这个数据直接的时间差）
（6）自旋时会适当放弃线程优先级之间的差异

释放偏向锁（偏向锁升级为轻量级锁）：开销大。执行步骤：
（1）等待全局安全点
（2）停止拥有锁的线程，遍历线程栈，存在锁记录的要修复记录和mark word，置为无锁状态
（3）唤醒线程，升级锁

获取轻量级锁：线程尝试获取锁，如果锁对象的mark word对应的线程还在占用则修改mark word的，修改重量级锁的标志位，代表升级为重量级锁；如果对应的线程没有占用则表示获取锁成功。

释放轻量级锁：释放的时候查看mark word与获取锁时拷贝的mark word值是否一致，如果不一致说明有其他线程尝试来竞争锁，则需要升级为重量级锁。此时需要解决一个问题，mark word的值被修改了，但是修改这个值的线程没有占用，则判断一下mark word对应的线程是否持有锁，如果没有，通过CAS修改值。

全局安全点（在这个时间点上没有字节码正在执行）

锁粗化：循环时加锁，例如循环中使用StringBuffer.append()方法，会造成一直加锁释放锁（反复进出临界区），效率低下。

ReentrantLock

ReentrantLock底层依赖AQS（AbstractQueuedSynchronizer），AQS底层依赖CAS操作。

关注ReentrantLock的lock和unlock方法。

ReentrantLock有两个锁的选择：FairSync（公平锁）和NonfairSync（非公平锁），默认使用非公平锁，非公平锁的吞吐量更高。两个锁的lock最终都是用到AQS的acquire方法

tryAcquire方法由两个锁各自实现，区别在于公平锁每次都是由队列的头部元素去获取锁，所以多了一个队头的判断。

addWaiter方法将线程加入等待队列尾部。

acquireQueued方法自旋获取资源，返回是否中断过（中断过表示线程在等待过程中休息被唤醒过，前置结点可能释放资源了）。如果自旋获取失败则直接出队（防止出现线程饥饿）。shouldParkAfterFailedAcquire方法获取前置结点是否有signal状态（这个状态代表释放资源后会通知后继结点），parkAndCheckInterrupt方法获取线程是否被中断过。

acquire的selfInterrupt方法表示中断线程，唤醒线程执行代码。为何要判断历史是否中断过，如果没有中断过，说明没有被唤醒执行，说明历史上一直在排队还没有获取过cpu资源（挺惨的）。如果被中断过说明已经被中断唤醒，已经在running了，不能去中断执行。

tryRelease是子类自行实现的，父类直接抛出了异常，因为有些子类并不需要这些方法，不需要实现，所以需要的子类自行实现即可。ReentrantLock内部类Sync继承Aqs，实现了上述的代码，这个对于公平锁和非公平锁都是一样的，所以不是在NonfairSync和FairSync去实现（锁的时候在两个子类去实现是因为一个是竞争，一个是排队。释放的时候只能一个一个串行化的释放，所以是一样的，在父类Sync实现即可）。

注：这里的中断不是指线程正在运行然后被中断运行，而是指被其他线程进行中断操作请求响应处理。例如a线程给b线程发送了中断的指令请求响应，此时b线程收到后保存上下文的信息，响应中断，执行指令。
